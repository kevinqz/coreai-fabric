# Redteam Report — "coreai-fabric: from model catalog to block+technique registry"

## 1. Verdict per layer

| Layer | Verdict | Why (one line) |
|---|---|---|
| **A — blocks registry** | **NEEDS REDESIGN** | The standalone normalized entity (SOLVED status, per-block Gate-B number, stored `used_by`, architecture-only key) is falsified on every axis; the surviving form is an optional `blocks:` array on the recipe schema plus envelope/provenance/license axes — a different, much smaller design. |
| **B — techniques registry** | **KILL** | "Auto-apply per block" has no execution vehicle (fabric refuses by design to drive the scripts where techniques live as source edits), triggers are framework-keyed not block-keyed, and the playbook already IS the curated ACE playbook — techniques.yaml is a third trailing prose copy wearing a schema tax. |
| **C — analyze** | **NEEDS REDESIGN** | Config-derived decomposition works on ~0 of the 35 multi-block upstreams it targets and the flagship repo's configs actively conceal the MoT; salvageable only as exact-match lookup + shape prefill + loud refusals — never "SOLVED" and never a coverage percentage. |
| **D — best-lane per block** | **NEEDS REDESIGN** | Per-block Gate-B numbers do not exist, the composite numbers never reach the catalog (`_EVAL_KEYS` drops them), and "the Gate-B number" is four incommensurable quantities under unrecorded protocols — ranking over it systematically crowns the weakest verification. |
| **Harness loops** | **NEEDS REDESIGN** | Failed exports leave zero structured trace (and fabric never sees 36/53 script-tool runs), the evaluator is negotiable by the same pair that measures, and coverage% is a self-defined signal a matcher inflates at zero cost — the loop's data substrate and its integrity rules must be built first. |

## 2. Confirmed findings (most severe first) and what the design MUST do

### FATAL

**F1. The flagship SenseNova decomposition is a false-SOLVED generator** *(A, C, D)*
Both "SOLVED" cells name lanes never built at those shapes: SigLIP exists only fused inside VLA encoders with a composite gate (no per-block cosine anywhere; the "same encoder lane" is a lingbot ViT / EuroBERT text encoder, not SigLIP, and never at 980px/patch-14); "qwen2-7b-lm via EO-1" conflates a fixed-shape non-AR denoise forward with an AR KV-cache generator, in MoT form ~14.5B dual-expert params the stock lane cannot ingest — and the 7B stock recipe is itself an unmeasured draft. The configs analyze would read are a stub + textbook Qwen2 with zero MoT trace against a 29.2GB checkpoint; the truth came from the paper and modeling code.
**MUST:** the design's own showcase must be relabeled NET-NEW across the board. Any "SOLVED" claim requires a built artifact at the claimed (architecture, shape, precision) point with its own measured number. Add the cheap weight-bytes-vs-config-implied-params cross-check as a mandatory tripwire.

**F2. "The Gate-B number" is at least four mathematically different quantities under unrecorded protocols** *(D, A)*
Match-rate vs cosine (bars 0.85 vs 0.999); per-row 14-dim min vs flattened 1600-dim cosine (structurally inflated); synthetic vs recorded inputs (with an in-repo 0.02-cosine catastrophe precedent, and the schema falsely claiming recorded inputs); fp16 vs fp32 reference; a near-zero waiver that marks passed below the bar; `_fidelity_tier` collapsing any pass to `high_fidelity`. None of the signature (n_obs, seed, protocol, dtype, granularity, boundary) has a schema home; reports die in gitignored `build/`.
**MUST:** metric identity includes protocol. Leaderboards exist only within (metric, granularity, input-protocol, reference-dtype, graph-boundary) cells, ranked on raw values with waiver flags surfaced. Extending the `gate_b` schema and the register whitelist is a forced **prerequisite**, not an implementation detail.

**F3. Layer C's target population defeats config reading at a measured base rate of ~0** *(C, A, D, loops)*
The 18 standard-config upstreams are single-block lanes where decomposition is a no-op (scaffold already handles them); the 24 LeRobot repos carry family+shapes but never the decomposition (pi0's own driver says shape is "UNCERTAIN until probed"); 11 defeat config reading outright (model.pt-only, trust_remote_code, stub configs, custom formats). Metadata is documented-unreliable in this exact corpus (pipeline_tag hand-corrected twice). No prediction/outcome logging exists, so matcher precision is unknowable and can never improve. A coverage% is "fabric never fakes a parity number" violated one level up.
**MUST:** ship only the honest reduced form — LeRobot parser (type → driver family, verbatim shape prefill), transformers exact-match on `architectures[0]` PLUS a proven size/shape envelope, hard tripwires forcing "MANUAL ANALYSIS REQUIRED" on auto_map/stub/gated/weight-bytes contradictions. Output vocabulary: "candidate lane (verify against modeling code)". Log every match against its eventual Gate-B outcome or don't emit matches.

**F4. Layer B's auto-apply has no execution vehicle** *(B, C)*
Techniques are trace-time source edits and monkeypatches inside hand-vendored copies of arbitrary upstream torch code, run manually in two venvs; convert.py refuses by design to drive script tools; scaffold emits only YAML. The same trap appears as three different code shapes across drivers; the load-bearing interface is the causal prose. The repo already contains the predicted failure live: Tier-3 structured technique fields with zero Python consumers and zero recipes using them.
**MUST:** kill techniques.yaml. The playbook stays canonical prose; the executable canonical form stays the drivers. The only defensible structuring is trigger-phrase indexing *inside* the playbook (it already has it: "ordered by how often they bite"). Any future technique metadata must ship with its executable consumer in the same commit or not at all.

### MAJOR

**F5. Blocks are not shape-stable reusable units; the standalone registry is normalization overhead** *(A, C)*
Cross-architecture driver-file reuse happened twice in 53 recipes; 14/21 script tools serve exactly one recipe; block shape identity is per-checkpoint (three pi05 fine-tunes = three static-shape identities). What *does* transfer (3/3 among flow-matching siblings) is family-level trap knowledge via copy-adaptation — which a "siglip: SOLVED" row doesn't carry. Mandatory per-recipe block references create a ~20×53 consistency surface whose pointer value grep already returns.
**MUST:** degenerate Layer A to an **optional** `blocks:` string array on the existing recipe schema (~10-line diff); derive `used_by` by grep/generation, never store it. No mandatory references, no standalone entity.

**F6. Per-block Gate-B numbers cannot be honestly derived, and even composite values have no machine-readable home** *(A, D)*
All gates score the composed deployable graph; attributing whole-graph numbers to constituent blocks is attribution fraud. `_EVAL_KEYS` omits the numeric value — the live catalog holds no Gate-B number for any action lane. **MUST:** record deployable-lane Gate B with provenance (blocks associate with the graph they compose — no fabricated per-block cosine); extend `_EVAL_KEYS` to carry value/min/median/n_obs/threshold (~1-line class of fix); treat playbook §7 as the SotA table's existing prototype.

**F7. The evaluator is too weak and negotiable to drive a self-improvement loop** *(D, loops)*
The number rewards exporting less (head-only 0.99999999999998 vs whole-LM 0.99948); every gaming knob is a free CLI parameter; gate relaxations landed in the same unreviewed commit as the results they enabled; pi0fast-base shipped published at 0.925/n_obs=2 against a 0.85 bar with only "passed" surfacing. **MUST:** SotA only within full protocol cells (F2); frozen held-out probe sets; a hard rule that gate definitions cannot change in the same change that flips a result to passed; fidelity tiers must not collapse pass→high_fidelity.

**F8. Weakness mining has no data substrate** *(loops, B)*
Failed converts write no manifest; no `failed` status exists; verify writes status only on pass; build/ is gitignored; the one CLI choke point never sees the 36/53 script-tool runs. **MUST:** build the substrate first — committed structured attempt records including failures with error signatures, a `failed/blocked` status, and run capture that works for manually-invoked script tools (e.g., a wrapper the printed invocation routes through).

**F9. Architecture-keyed block identity has no license/provenance axis** *(A, D, C, B)*
The same SigLIP block exists under apache-2.0, Gemma terms, and CC-BY-NC; the repo already holds the pi0-base contradiction in published form; every enforcement point reads one `upstream.license` per recipe. "Best lane" would compare different weight sets and crown winners with no legal artifact. **MUST:** block entries carry per-instance provenance (upstream_repo, revision, tensor-slice) + license posture as separate fields; analyze's per-block identifications must **feed triage_license**, not just lane selection; SotA records carry an artifact-availability flag.

**F10. No file-level content gate on the publish or git path for block-slice weights** *(A, B, C, loops)*
`upload_folder` ships bundle dirs recursively with no content allowlist; extracted raw sub-checkpoints of restricted upstreams sit one directory convention away; committed per-block fixtures would be derivative data of NC weights with no gate on `git push`. **MUST:** bundle content allowlist + per-file provenance on publish; an explicit policy (and .gitignore/CI guard) for any fixtures/blocks directory; index-only blocks explicitly excluded from any regression battery rather than silently skipped.

**F11. Technique triggers are framework-keyed, and mot-split violates an unstated bounded-forward-pass precondition Gate B cannot detect** *(B, loops)*
The bool-mask monkeypatch is a lerobot artifact (triplicated in exactly the three lerobot lanes, absent elsewhere); eager-vs-SDPA splits by codebase, not block family. Graph-split is valid only under a bounded host loop (4–10 denoise steps); an AR generator reloading N multi-GB programs per token passes cosine parity while being undeployable. **MUST:** any technique record carries framework scope + explicit preconditions (e.g., "host loop pass count bounded and small"); add a deployability facet distinct from cosine parity.

**F12. Block facts are points in (graph, shape, precision, host-contract) space and the 0x10004 constraint is per-program compositional** *(A, C, D, loops)*
Same MoE stack loads at L=12 fp16, fails at L=18; int8 flips the gate; a model 100% covered by individually SOLVED blocks can be unloadable as composed. **MUST:** key every block fact to an envelope tuple; scope any coverage claim to attempted envelopes; loadability is gated per composed bundle, never inferred from block-level facts.

**F13. The two-repo mirrored-vocabulary contract is already broken in both directions** *(A, B, C, D)*
Fabric's schema cites a catalog capabilities authority that doesn't exist; fabric's traits enum hard-fails catalog validation (empirically demonstrated); fabric's test fixture validates against a fictional catalog schema; four schema-first catalog commits each rate-limited a lane. **MUST:** one authority per vocabulary, no mirrors — canonical registry fabric-side, catalog gets a batched derived facet; fix or delete the fictional test fixture; reconcile traits/bundle_kind drift before adding any new vocabulary.

**F14. Stored per-block `used_by` lists deterministically reintroduce the PR #25–28 merge-collision class, worse** *(A, loops)*
Two in-flight registrations of SigLIP-bearing models conflict on the same block entry lines (empirically verified), and mid-entry list mutations are not union-resolvable like EOF appends. **MUST:** reverse indexes are derived, never stored (same fix as F5); keep the block entity out of the catalog or budget the full 7th-entity surface (contribute/validate/audit/counts/generate).

### MINOR

**F15. Registry freshness cannot be machine-checked against a moving Apple toolchain** *(A, B, C)*
CI runs no conversion; every `min_tool_version` bump silently ages SOLVED claims. **MUST:** verification timestamps + toolchain version on every registry fact, and a CI invariant coupling recipes to block references (the repo's atomic-commit discipline handles the rest).

**F16. Zero-regression validation needs scoping, not abandonment** *(loops, A, B, D)*
Full-fleet Gate-B re-verification per registry change is unaffordable, but the cheap evaluator already exists in committed form (scratchpad random-weight smoke scripts for MoE lowering, graph-split chaining, SDPA masks). **MUST:** promote the smokes to a registry-keyed local proxy battery with an explicit exclusion for scale-dependent effects (0x10004, fp16 load ceilings); drop the "zero-regression" language otherwise.

## 3. Strongest surviving alternatives (scope-alternatives lens)

1. **Optional `blocks:` string array on the recipe schema** (~10-line diff, derived `used_by` via grep/generation). Best value/cost of the entire proposal; replaces Layer A.
2. **Playbook §7 as the SotA table** + a bounded `register.py` extension carrying numeric Gate-B values and the protocol signature (F2's cell key) into the existing catalog `evaluation` block. Replaces Layer D's new entity.
3. **Reduced analyze**: LeRobot config parser (family → driver, verbatim shape prefill), `architectures[0]` exact-match within a proven size/shape envelope, hard refusal tripwires, weight-bytes cross-check. Replaces Layer C.
4. **Failure substrate before any loop**: failure manifests + `failed` status + script-tool run capture; promote the existing smoke scripts to the loop's cheap evaluator. Replaces the loops layer's launch dependency.
5. **Keep the playbook as the ACE context** — it already has numbered entries, trigger phrases, frequency ordering, and a working human+agent reflector/curator. Replaces Layer B entirely.

## 4. Open design questions (decision-shaped)

1. **Block identity key:** architecture alone, or (architecture, framework, envelope tuple, weight provenance)? At what point of key-widening do you accept degeneration to per-recipe facts and stop?
2. **Vocabulary authority:** which single repo owns block ids and evaluation vocabulary, and is the catalog's role strictly a batched derived facet generated from fabric? (Requires deciding the fate of the fictional test fixture and the traits enum.)
3. **Protocol signature schema:** exactly which fields (n_obs, seed, input-protocol, reference-dtype, granularity, graph-boundary, waiver flags) enter `gate_b` and `_EVAL_KEYS`, and does the catalog reject evaluations missing them?
4. **Failure capture mechanism:** given fabric refuses to drive 36/53 script tools, what is the concrete hook that records failed attempts — a mandatory wrapper around the printed invocation, a manifest the driver itself writes, or something else?
5. **License routing:** does analyze emit per-block license findings that feed `triage_license` and the publish gate, and what file-level control keeps extracted block weights out of both HF bundles and git commits?
6. **Anchor namespace:** where do blocks proven only by draft/index-only recipes live — fabric recipe ids (full 53-id namespace) with a structured measured-parity field, or a new catalog availability state? Pick one; the current catalog rejects both.

## 5. Refuted-findings appendix

- **"Loop speed vs registry reach dilemma (human PR merge in the loop)"** — rejected: registration is already asynchronous and non-blocking by shipped design (fabric never gates on `registered`), catalog publication is batched by operating norm, and fabric-authored structured vocabulary demonstrably flows to the catalog's programmatic surface; what survives is a one-paragraph spec note (canonical registry fabric-side, loop acceptance never blocks on catalog merge), not an architectural dilemma.
